## Overview

This repository (`openshift/release`) holds:
- OpenShift cluster manifests
- Component build manifests
- **CI workflow configurations** for OpenShift component repositories (both OKD and OCP)

The CI system is powered by:
- **Prow**: Kubernetes-based CI/CD system
- **ci-operator**: Tool that orchestrates builds and tests
- **Step Registry**: Library of reusable test components

## How the CI System Works

### The Complete Flow

```
┌─────────────────────────────────────────────────────────────────┐
│  Developer Code (Your GitHub Repository)                        │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│  CI Config Files (ci-operator/config/)                          │
│  ├── You EDIT these YAML files                                  │
│  └── Define: builds, images, tests, releases                    │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼  make update
┌─────────────────────────────────────────────────────────────────┐
│  Prow Job Definitions (ci-operator/jobs/)                       │
│  ├── AUTO-GENERATED - Do NOT edit manually                      │
│  └── Created by ci-operator-prowgen tool                        │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼  On PR/Merge/Schedule
┌─────────────────────────────────────────────────────────────────┐
│  Prow Triggers the Job                                          │
│  ├── PR events (presubmit)                                      │
│  ├── Merge events (postsubmit)                                  │
│  └── Scheduled (periodic)                                       │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│  ci-operator Executes                                           │
│  ├── Builds images                                              │
│  ├── Provisions test clusters                                   │
│  ├── Runs tests                                                 │
│  └── Collects results                                           │
└────────────────────────┬────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│  Results Displayed in Prow UI                                   │
│  └── https://prow.ci.openshift.org                              │
└─────────────────────────────────────────────────────────────────┘
```

### Key Principle

**You only edit files in `ci-operator/config/`**. Everything else is auto-generated by `make update`.

## Repository Structure

```
openshift/release/
├── ci-operator/
│   ├── config/                    # ← YOU EDIT THESE
│   │   └── <org>/<repo>/
│   │       ├── <org>-<repo>-<branch>.yaml
│   │       └── <org>-<repo>-<branch>__periodics.yaml
│   │
│   ├── jobs/                      # ← AUTO-GENERATED
│   │   └── <org>/<repo>/
│   │       └── <org>-<repo>-<branch>-<type>.yaml
│   │
│   ├── step-registry/             # ← REUSABLE COMPONENTS
│   │   ├── <step-name>/
│   │   │   ├── <step-name>-ref.yaml
│   │   │   └── <step-name>-commands.sh
│   │   ├── <workflow-name>/
│   │   │   └── <workflow-name>-workflow.yaml
│   │   └── <chain-name>/
│   │       └── <chain-name>-chain.yaml
│   │
│   └── templates/                 # ← LEGACY (deprecated)
│
├── core-services/                 # Core service configs
│   ├── prow/                      # Prow configuration
│   └── release-controller/        # Release controller configs
│
├── services/                      # Additional service configs
├── tools/                         # Container image builds for tooling
└── Makefile                       # Build and validation targets
```

### Key Directories

| Directory | Purpose | Edit Manually? |
|-----------|---------|----------------|
| `ci-operator/config/` | CI configuration (source of truth) | **YES** ✅ |
| `ci-operator/jobs/` | Generated Prow jobs | **NO** ❌ |
| `ci-operator/step-registry/` | Reusable test components | **YES** ✅ |
| `core-services/` | Core service configurations | **YES** ✅ |

## Understanding CI Configuration Files

### File Naming Convention

```
ci-operator/config/<org>/<repo>/<org>-<repo>-<branch>.yaml
```

**Examples:**
- `ci-operator/config/openshift-pipelines/performance/openshift-pipelines-performance-main.yaml`
- `ci-operator/config/openshift/installer/openshift-installer-release-4.18.yaml`

### Configuration File Structure

Every CI configuration file has the following sections:

```yaml
# 1. BUILD ENVIRONMENT
build_root:
  image_stream_tag:
    name: builder
    namespace: ocp
    tag: rhel-9-golang-1.22-openshift-4.18

# 2. CUSTOM IMAGES (optional)
images:
  items:
  - dockerfile_path: Dockerfile
    to: my-image

# 3. RELEASES (OpenShift version to test)
releases:
  latest:
    integration:
      name: "4.18"
      namespace: ocp

# 4. RESOURCE LIMITS
resources:
  '*':
    requests:
      cpu: 500m
      memory: 1Gi

# 5. TEST DEFINITIONS
tests:
- as: unit-test
  commands: make test
  container:
    from: src

# 6. METADATA (auto-generated)
zz_generated_metadata:
  branch: main
  org: openshift-pipelines
  repo: performance
```

## Example: Performance Testing Configuration

Let's break down the `openshift-pipelines/performance` configuration:

### Build Root & Custom Image

```yaml
build_root:
  image_stream_tag:
    name: release
    namespace: openshift
    tag: rhel-8-release-golang-1.18-openshift-4.12

images:
  items:
  - dockerfile_literal: |-
      FROM src
      # Install performance testing tools
      RUN yum install -y httpd-tools parallel
      RUN curl -Lso /usr/local/bin/jq https://github.com/stedolan/jq/releases/download/jq-1.7.1/jq-linux64
      RUN curl -Lso /usr/local/bin/kubectl https://storage.googleapis.com/.../kubectl
    from: src
    to: openshift-pipelines-performance-runner
```

**Purpose**: Creates a custom container image with performance testing tools (jq, kubectl, httpd-tools, etc.)

### OpenShift Release

```yaml
releases:
  latest:
    release:
      architecture: amd64
      channel: fast
      version: "4.20"
```

**Purpose**: Tests will run against OpenShift 4.20 (fast channel)

### Test Definitions

#### Periodic Test (Scheduled)

```yaml
tests:
- as: max-concurrency-downstream-nightly
  cron: 0 1 * * 2,5  # Runs at 1 AM on Tuesday and Friday
  steps:
    cluster_profile: aws-pipelines-performance
    env:
      DEPLOYMENT_TYPE: downstream
      NIGHTLY_BUILD: "true"
      TEST_NAMESPACE: "5"
      TEST_SCENARIO: math
      TEST_SCENARIOS: 200/12 200/14 200/16 200/18 200/20
    workflow: openshift-pipelines-max-concurrency
```

**Breakdown:**
- `as`: Job name (shows in Prow UI)
- `cron`: Schedule in cron format
- `cluster_profile`: Dedicated AWS account/resources for performance testing
- `env`: Environment variables passed to the test
- `workflow`: Reusable workflow from step-registry

#### Manual/Optional Test

```yaml
- always_run: false
  as: max-concurrency-downstream-1-12-1000-100
  optional: true
  steps:
    cluster_profile: aws-pipelines-performance
    env:
      DEPLOYMENT_VERSION: "1.13"
      TEST_SCENARIOS: 1000/100
    workflow: openshift-pipelines-max-concurrency
```

**Breakdown:**
- `always_run: false`: Won't run automatically on PRs
- `optional: true`: Can be manually triggered
- Useful for: Ad-hoc testing, experiments, one-time validations

### Test Scenarios

The performance config tests various configurations:

| Test Type | Purpose | Example Env Vars |
|-----------|---------|------------------|
| **Max Concurrency** | Test pipeline throughput | `TEST_SCENARIOS: 200/12` (200 tasks, 12 concurrent) |
| **High Availability** | Test with multiple replicas | `DEPLOYMENT_PIPELINES_CONTROLLER_HA_REPLICAS: "10"` |
| **QPS/Burst Tuning** | Test API rate limits | `DEPLOYMENT_PIPELINES_KUBE_API_QPS: "50"` |
| **Version Testing** | Test specific versions | `DEPLOYMENT_VERSION: "1.20"` |
| **Signing Performance** | Test Tekton Chains | `TEST_SCENARIO: signing-tr-tekton-bigbang` |

## Creating New CI Configuration from Scratch

### Method 1: Interactive Tool (Recommended for Beginners)

```bash
cd /path/to/openshift/release
make new-repo
```

The tool will prompt you for:
1. Organization (e.g., `openshift-pipelines`)
2. Repository name (e.g., `new-feature`)
3. Branch name (e.g., `main`)

It will automatically:
- Create the config directory
- Generate a basic configuration
- Run `make update` to generate Prow jobs

### Method 2: Manual Creation (More Control)

#### Step 1: Create Directory Structure

```bash
mkdir -p ci-operator/config/openshift-pipelines/new-feature
```

#### Step 2: Create Configuration File

Create `ci-operator/config/openshift-pipelines/new-feature/openshift-pipelines-new-feature-main.yaml`:

```yaml
# Build environment - base image with tools
build_root:
  image_stream_tag:
    name: builder
    namespace: ocp
    tag: rhel-9-golang-1.22-openshift-4.18

# Optional: Build custom images from your repository
images:
  items:
  - dockerfile_path: Dockerfile  # Path in your repo
    to: my-test-image

# OpenShift release to test against
releases:
  latest:
    integration:
      name: "4.18"
      namespace: ocp

# Resource constraints
resources:
  '*':
    requests:
      cpu: 500m
      memory: 1Gi

# Test definitions
tests:
# Unit tests - run on every PR
- as: unit-tests
  commands: make test-unit
  container:
    from: src  # Run in source code container

# Integration tests - run on every PR
- as: integration
  commands: make test-integration
  container:
    from: src

# E2E test on AWS - run on every PR
- as: e2e-aws
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws

# Nightly E2E test - scheduled
- as: nightly-e2e
  cron: "0 2 * * *"  # 2 AM daily
  steps:
    cluster_profile: aws
    env:
      TEST_TYPE: extended
    workflow: openshift-e2e-aws

# Optional manual test
- always_run: false
  as: performance-test
  optional: true
  steps:
    cluster_profile: aws-pipelines-performance
    env:
      LOAD_LEVEL: high
    workflow: custom-perf-workflow
```

#### Step 3: Generate Prow Jobs

```bash
make update
```

This command runs multiple targets:
- `make ci-operator-config` - Validates and normalizes config
- `make jobs` - Generates Prow job definitions
- `make prow-config` - Updates Prow configuration
- `make registry-metadata` - Updates step registry metadata
- `make release-controllers` - Updates release controller configs
- `make boskos-config` - Updates resource pool configs

#### Step 4: Validate

```bash
make checkconfig
```

Look for any errors in the output. Common issues:
- Invalid YAML syntax
- Missing required fields
- Invalid workflow references
- Resource limit issues

#### Step 5: Commit and Create PR

```bash
git checkout -b add-new-feature-ci
git add ci-operator/config/openshift-pipelines/new-feature/
git add ci-operator/jobs/openshift-pipelines/new-feature/
git commit -m "Add CI configuration for openshift-pipelines/new-feature

This adds:
- Unit tests
- Integration tests
- E2E tests on AWS
- Nightly extended testing
"
git push origin add-new-feature-ci
```

Then create a pull request on GitHub.

## Advanced Topics

### Using Existing Workflows from Step Registry

Instead of defining all test steps, use pre-built workflows:

```bash
# Search for workflows
/step-finder aws upgrade workflow
/step-finder install operator step
```

Common workflows:

| Workflow | Purpose | Usage |
|----------|---------|-------|
| `openshift-e2e-aws` | Basic AWS e2e tests | General testing |
| `ipi-aws` | AWS IPI cluster provisioning | Custom multi-stage tests |
| `openshift-upgrade` | Upgrade testing | Testing upgrades |
| `generic-claim` | Use pre-provisioned cluster | When you have dedicated clusters |
| `openshift-pipelines-max-concurrency` | Performance testing | Load/performance tests |

**Example using a workflow:**

```yaml
tests:
- as: e2e-upgrade
  steps:
    cluster_profile: aws
    workflow: openshift-upgrade-aws
```

### Creating Custom Workflows

When existing workflows don't fit your needs:

#### 1. Create Workflow Directory

```bash
mkdir -p ci-operator/step-registry/openshift-pipelines/my-workflow
```

#### 2. Create Workflow Definition

File: `ci-operator/step-registry/openshift-pipelines/my-workflow/openshift-pipelines-my-workflow-workflow.yaml`

```yaml
workflow:
  as: openshift-pipelines-my-workflow
  documentation: |-
    This workflow tests OpenShift Pipelines with custom configuration.
  steps:
    pre:
    - ref: ipi-install-hosted-loki
    - ref: ipi-install-rbac
    test:
    - ref: openshift-pipelines-my-test
    post:
    - ref: gather-must-gather
    - ref: openshift-pipelines-gather-performance
```

#### 3. Create Custom Test Step

File: `ci-operator/step-registry/openshift-pipelines/my-test/openshift-pipelines-my-test-ref.yaml`

```yaml
ref:
  as: openshift-pipelines-my-test
  documentation: |-
    Runs custom OpenShift Pipelines tests.
  commands: openshift-pipelines-my-test-commands.sh
  from: openshift-pipelines-test-runner
  resources:
    requests:
      cpu: 100m
      memory: 200Mi
  timeout: 1h0m0s
```

#### 4. Create Test Script

File: `ci-operator/step-registry/openshift-pipelines/my-test/openshift-pipelines-my-test-commands.sh`

```bash
#!/bin/bash
set -euo pipefail

echo "Starting custom OpenShift Pipelines tests..."

# Your test logic here
make test-custom

# Example: Check environment variables
if [[ "${CUSTOM_PARAM:-}" != "" ]]; then
    echo "Custom parameter: ${CUSTOM_PARAM}"
fi

echo "Tests completed successfully!"
```

Make it executable:
```bash
chmod +x ci-operator/step-registry/openshift-pipelines/my-test/openshift-pipelines-my-test-commands.sh
```

#### 5. Update Metadata

```bash
make registry-metadata
```

#### 6. Use Your Custom Workflow

```yaml
tests:
- as: custom-test
  steps:
    cluster_profile: aws
    env:
      CUSTOM_PARAM: "my-value"
    workflow: openshift-pipelines-my-workflow
```

### Security Best Practices for Step Scripts

When writing test scripts, protect sensitive information:

```bash
#!/bin/bash

# CRITICAL: Disable tracing when handling sensitive data
[[ $- == *x* ]] && WAS_TRACING=true || WAS_TRACING=false
set +x  # Disable tracing

# Handle passwords, tokens, or API keys
SECRET=$(oc get secret my-secret --template='{{.data.password}}' | base64 -d)
echo "$SECRET" > "${SHARED_DIR}/password"

# Restore previous tracing state
$WAS_TRACING && set -x

# Continue with non-sensitive operations
echo "Configuration complete"
```

**Key points:**
- Never `echo` passwords, tokens, or credentials
- Be careful with `set -x` (it logs all commands)
- Use `${SHARED_DIR}` to pass data between steps
- Keep tracing disabled for minimal scope

### Multi-Stage Test Configuration

For complex testing scenarios:

```yaml
tests:
- as: complex-e2e
  steps:
    cluster_profile: aws
    env:
      CUSTOM_VAR: "value"
    pre:
    - ref: ipi-install-hosted-loki       # Setup logging
    - ref: my-custom-setup                # Your setup step
    test:
    - ref: my-first-test
    - ref: my-second-test
    post:
    - ref: gather-must-gather             # Always runs
    - ref: my-custom-cleanup              # Your cleanup
```

### Cluster Profiles

Different cloud providers and resource pools:

| Profile | Cloud | Purpose |
|---------|-------|---------|
| `aws` | AWS | General AWS testing |
| `aws-2` | AWS | Second AWS account |
| `aws-pipelines-performance` | AWS | Dedicated performance testing |
| `gcp` | GCP | Google Cloud testing |
| `azure4` | Azure | Azure testing |
| `vsphere-elastic` | vSphere | VMware testing |
| `packet` | Equinix Metal | Bare metal testing |

### Variant Periodic Configurations

For release-specific periodic testing:

**File**: `ci-operator/config/openshift-pipelines/operator/openshift-pipelines-operator-release-4.18__periodics.yaml`

```yaml
base_images:
  # Copy from main config
  
build_root:
  # Copy from main config

releases:
  latest:
    release:
      version: "4.18"

tests:
# ONLY periodic tests with interval or cron
- as: nightly-e2e
  interval: 24h
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws

- as: weekly-upgrade
  cron: "0 2 * * 0"  # Sunday 2 AM
  steps:
    cluster_profile: aws
    workflow: openshift-upgrade-aws
```

Then run `make update` - jobs go to separate `-periodics.yaml` in `ci-operator/jobs/`.

## Common Patterns and Examples

### Pattern 1: Simple Unit Tests Only

```yaml
build_root:
  image_stream_tag:
    name: builder
    namespace: ocp
    tag: rhel-9-golang-1.22-openshift-4.18

resources:
  '*':
    requests:
      cpu: 500m
      memory: 1Gi

tests:
- as: unit
  commands: make test-unit
  container:
    from: src

- as: lint
  commands: make lint
  container:
    from: src

- as: verify
  commands: make verify
  container:
    from: src
```

### Pattern 2: Build Image and Run E2E Tests

```yaml
build_root:
  image_stream_tag:
    name: builder
    namespace: ocp
    tag: rhel-9-golang-1.22-openshift-4.18

images:
  items:
  - dockerfile_path: Dockerfile
    to: my-operator

releases:
  latest:
    integration:
      name: "4.18"
      namespace: ocp

tests:
- as: unit
  commands: make test-unit
  container:
    from: src

- as: e2e-aws
  steps:
    cluster_profile: aws
    test:
    - as: install-operator
      cli: latest
      commands: |
        make install
        oc wait --for=condition=available --timeout=5m deployment/my-operator
      from: my-operator
      resources:
        requests:
          cpu: 100m
    - as: run-tests
      cli: latest
      commands: make test-e2e
      from: my-operator
      resources:
        requests:
          cpu: 100m
    workflow: ipi-aws
```

### Pattern 3: Multi-Version Testing

```yaml
releases:
  latest:
    candidate:
      product: ocp
      version: "4.18"

tests:
# Test on 4.18
- as: e2e-4-18
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws

# Test on 4.17
- as: e2e-4-17
  releases:
    latest:
      candidate:
        product: ocp
        version: "4.17"
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws

# Test on 4.16
- as: e2e-4-16
  releases:
    latest:
      candidate:
        product: ocp
        version: "4.16"
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws
```

### Pattern 4: Periodic Performance Testing

```yaml
build_root:
  image_stream_tag:
    name: release
    namespace: openshift
    tag: rhel-9-release-golang-1.22-openshift-4.18

images:
  items:
  - dockerfile_literal: |-
      FROM src
      RUN yum install -y httpd-tools jq
      RUN curl -Lo /usr/local/bin/kubectl https://storage.googleapis.com/.../kubectl
    to: perf-runner

releases:
  latest:
    release:
      version: "4.18"

tests:
# Daily performance test
- as: perf-daily
  cron: "0 3 * * *"
  steps:
    cluster_profile: aws-pipelines-performance
    env:
      LOAD_LEVEL: standard
      DURATION: 1h
    workflow: my-perf-workflow

# Weekly high-load test
- as: perf-weekly-high
  cron: "0 2 * * 0"
  steps:
    cluster_profile: aws-pipelines-performance
    env:
      LOAD_LEVEL: high
      DURATION: 4h
    workflow: my-perf-workflow

# Manual stress test
- always_run: false
  as: perf-stress
  optional: true
  steps:
    cluster_profile: aws-pipelines-performance
    env:
      LOAD_LEVEL: extreme
      DURATION: 8h
    workflow: my-perf-workflow
  timeout: 10h0m0s
```

### Pattern 5: Using Cluster Claims (Pre-provisioned Clusters)

```yaml
tests:
- as: e2e-claim
  cluster_claim:
    architecture: amd64
    cloud: aws
    owner: pipelines
    product: ocp
    timeout: 1h0m0s
    version: "4.18"
  steps:
    test:
    - as: run-tests
      cli: latest
      commands: make test-e2e
      from: src
      resources:
        requests:
          cpu: 100m
    workflow: generic-claim
```

### Pattern 6: Skip Tests Based on File Changes

```yaml
tests:
- as: e2e-expensive
  skip_if_only_changed: ^(LICENSE|OWNERS|README\.md|\.gitignore)$|^docs/|^examples/
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws
```

This test will be skipped if ONLY documentation or non-code files changed.

```bash

make new-repo

# Validate and generate everything
make update

# Individual targets
make ci-operator-config     # Validate CI configs
```

### File Locations Quick Reference

| What | Where |
|------|-------|
| CI config (edit) | `ci-operator/config/<org>/<repo>/<org>-<repo>-<branch>.yaml` |

| Custom step | `ci-operator/step-registry/<name>/<name>-ref.yaml` |
| Step script | `ci-operator/step-registry/<name>/<name>-commands.sh` |

## Troubleshooting

### Common Issues and Solutions

#### Issue: `make update` fails with validation errors

```
Error: field tests[5].steps.workflow is not defined in step registry
```

**Solution**: The workflow you're referencing doesn't exist. Search for available workflows:
```bash
/step-finder <workflow-name>
```

#### Issue: Generated jobs are not in the expected location

**Solution**: Jobs are generated based on the config file location. Ensure:
```
ci-operator/config/openshift-pipelines/performance/openshift-pipelines-performance-main.yaml
```
Generates jobs in:
```
ci-operator/jobs/openshift-pipelines/performance/
```

#### Issue: Test doesn't run on PRs

**Causes:**
1. `always_run: false` is set
2. `optional: true` is set
3. `skip_if_only_changed` pattern matches

**Solution**: Remove those fields or adjust the skip pattern.

#### Issue: Cluster provisioning times out

**Solution**: Increase timeout in test definition:
```yaml
tests:
- as: e2e-slow
  timeout: 2h0m0s  # Default is 1h
  steps:
    cluster_profile: aws
    workflow: openshift-e2e-aws
```

#### Issue: Test fails to find custom image

**Solution**: Ensure the image is built and referenced correctly:
```yaml
images:
  items:
  - dockerfile_path: Dockerfile
    to: my-image  # ← This name

tests:
- as: test
  steps:
    test:
    - from: my-image  # ← Must match
      commands: make test
```

#### Issue: Step script not executable

```
Error: exec format error
```

**Solution**: Make the script executable and add shebang:
```bash
chmod +x ci-operator/step-registry/my-step/my-step-commands.sh
```

```bash
#!/bin/bash
set -euo pipefail
# Your script here
```

## Comparison: Performance Config vs. Standard Config

| **Aspect** | **Performance Config** | **Standard Config** |
|------------|------------------------|---------------------|
| **Purpose** | Load/performance testing | Functional testing |
| **Primary Test Type** | Periodic (cron-scheduled) | Presubmit (PR-triggered) |
| **Cluster Profile** | `aws-pipelines-performance` | `aws`, `gcp`, or `cluster-claim` |
| **Custom Build Image** | Yes (with perf tools) | Usually not needed |
| **Test Count** | 100+ variants | 2-10 tests |
| **Execution Frequency** | Daily/weekly | Every PR + periodic |
| **Resource Requirements** | High (dedicated resources) | Standard |
| **Typical Timeout** | 4-8 hours | 1-2 hours |
| **Result Usage** | Trend analysis, benchmarks | Pass/fail validation |

## Summary

1. **Edit only** `ci-operator/config/` files
2. **Run** `make update` to generate Prow jobs
3. **Validate** with `make checkconfig`
4. **Commit** both config and generated files
5. **Use** existing workflows from step-registry when possible
6. **Search** with `/step-finder` before creating new components

**Key Principle**: The configuration in `ci-operator/config/` is the source of truth. Everything else is generated or derived from it.
